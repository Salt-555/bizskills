# MVP Storefront

Deploys a storefront in one of two modes: **market-testing** (email capture, no Stripe) or **full** (Stripe checkout + download).

## Two-Path System

```
mvp-storefront
├── market-testing (demand validation)
│   ├── Landing page (ALLMIND design)
│   ├── Email capture form → KV (POST /signup)
│   ├── Signup count endpoint (GET /count)
│   ├── /privacy, /terms
│   ├── No Stripe, no webhook, no download
│   └── Deploys in seconds, no payment proxy needed
│
└── full (production storefront)
    ├── Everything from market-testing PLUS:
    ├── Stripe Payment Link checkout
    ├── Webhook → Resend download email
    ├── Success page with auto-download
    ├── Download endpoint (GET /download)
    └── Payment proxy via Service Binding
```

**Same URL the entire lifecycle.** Market-testing deploys first → ads run → if validated, redeploy as full on the same worker. Ads don't break, links don't change.

## Pipeline Position

```
idea-miner → mvp-storefront (market-testing) → ads-manager → execution-tracker
                                                          ↓ (validated)
                                    mvp-storefront (full) → mvp-builder → live
```

## When to Use Each Mode

**market-testing** — Default starting point for every new idea:
- "Deploy a storefront for X" (no mention of payments)
- "Test demand for this idea"
- "Set up a landing page to collect emails"
- Before building the MVP — validate interest first

**full** — After market-testing validates demand:
- "Flip to full storefront" / "Add payments"
- "This idea is validated, make it purchasable"
- After ads show good signup rates or CPA
- Only then: create Stripe product, build MVP

## What This Builds

### Market-Testing Mode
- Landing page (ALLMIND cream/gold/black design) with email capture form
- POST /signup — stores `{email, timestamp, source, utm_*}` in KV
- GET /count — returns signup count (for quick checks)
- Privacy policy + Terms of Service
- Cloudflare Worker deploy (signups KV only)

### Full Mode (Everything Above PLUS)
- Stripe Payment Link for checkout (no custom checkout handler)
- Webhook handler for payment confirmation + download email
- Post-checkout success page (auto-downloads product)
- Download endpoint (GET /download)
- Resend email delivery
- Payment proxy via Service Binding

## What This Does NOT Build
- The actual product/tool/app
- User dashboards or app logic
- Subscription management (one-time purchases only)

## Digital Product Download Flow

For downloadable products (zip packages, not SaaS), use this pattern:

**Email Collection: Stripe handles it. We don't.**
No email input on the landing page. Stripe collects email natively during checkout. The webhook gets the email from `session.customer_email` and sends the download link via Resend. One collection point, zero friction.

### Storage
- Package the product as a zip (< 25MB for KV)
- Embed as base64 in the worker JS (`const ZIP_DATA = "base64..."`)
- Or upload to KV namespace and serve from there

### Download Endpoint
```
GET /download?email={email}
```
Serves the zip file. The email param is for logging/tracking — the success page and Resend email generate this URL after verifying payment via Stripe session lookup. For simple products, the download doesn't need KV verification — the URL is only given to verified purchasers.

### Checkout Flow (Payment Link)
The landing page has `<a>` tags linking directly to a Stripe Payment Link:
```
<a href="{{PAYMENT_LINK_URL}}" class="purchase-btn">{{CTA_TEXT}}</a>
```
No JS, no fetch call, no custom checkout handler. Stripe handles everything: email collection, payment, receipt.

**Create the Payment Link via Stripe API:**
- Use the price ID from the deploy output
- Set `after_completion` type to `redirect`
- Set redirect URL to `https://{script-name}.yourdomain.com/success?session_id={CHECKOUT_SESSION_ID}`

**CRITICAL: Set `customer_creation: 'always'` on the Payment Link.**
Without this, Stripe collects the email but doesn't create a customer. The session has `customer_email: null` and `customer: null`, so the success page can't get the email and the webhook can't send the download email. Always set this when creating the Payment Link:
```
customer_creation: 'always'
```
- **MUST set `customer_creation: 'always'`** (see pitfall below)

### Payment Link Pitfall: customer_creation MUST Be 'always'
The default Payment Link `customer_creation` is `'if_required'`. This means:
- Stripe collects email in the checkout form
- But does NOT create a Stripe customer
- `session.customer` = null
- `session.customer_email` = null
- Success page session lookup fails → shows manual email entry
- Webhook can't find email → can't send download email
- **Everything breaks silently**

Always create Payment Links with `customer_creation: 'always'`. If you forget, you must create a new Payment Link — this setting cannot be changed after creation.

### Email Recovery (Customer Lookup)
The `/session` proxy endpoint looks up email from Stripe:
1. Fetch checkout session by ID
2. Try `session.customer_email`
3. If null and `session.customer` exists → fetch `GET /customers/{session.customer}` → use `customer.email`
This is used by the success page and the webhook handler.

### Webhook Email
After Stripe Checkout completes:
- Webhook fires `checkout.session.completed`
- Email comes from `session.customer_email` (set by Stripe during checkout)
- If email is null, fallback to customer lookup via proxy `/session`
- Resend email sent with download link

**PITFALL:** `sendDownloadEmail()` must be called OUTSIDE any `if (env.SALES_TRACKING)` block. If placed inside, the email only sends when SALES_TRACKING KV is bound — otherwise no email fires. The call should be at root level of the `checkout.session.completed` handler, not nested.

### Success Page
After Stripe Checkout redirects to `/success?session_id=cs_...`:
- Worker fetches session from Stripe via proxy `/session` endpoint to get customer email
- Page renders with download button linking to `/download?email={email}`
- If session lookup fails (network issue), fallback: manual email entry field
- "A copy has also been sent to your email" sets expectation for the Resend email

### Download Flow Summary
1. Customer clicks "Get Access" → lands on Stripe Payment Link checkout
2. Stripe collects email + payment → redirects to `/success?session_id=cs_...`
3. Worker calls proxy `/session` → gets email from Stripe → renders success page with `/download?email=...` link (auto-triggers download after 1.5s)
4. Webhook fires `checkout.session.completed` → looks up customer for email → sends Resend email with download link
5. Customer downloads (auto from page or from email link) → `GET /download?email=...` → serves zip

**After storefront deploy:** Hand off to mvp-builder for the actual product. Do NOT continue building product logic in this skill. Storefront = landing page + Stripe + legal. Product = separate skill (mvp-builder).

## Architecture
**Payment Link + Proxy Pattern** — Stripe Payment Link handles checkout. Proxy handles Stripe API calls. Workers handle delivery.
- Deploy: Cloudflare REST API (NOT wrangler). Token in env as CLOUDFLARE_TOKEN.
- Account ID: YOUR_CF_ACCOUNT_ID
- Payments: Stripe Payment Link (created via API)
- Proxy: Service Binding → `payment-proxy` (for session lookup, not checkout)
- Frontend: plain HTML/CSS/JS inline in Worker
- Email: Resend for download delivery via webhook

### Why Payment Link?
- No custom checkout handler needed — Stripe hosts the checkout
- Landing page is just `<a href="PAYMENT_LINK_URL">` — zero JS
- Stripe handles email collection, payment, receipt
- We only handle: success page + download delivery + webhook email

### Why Proxy?
- Holds Stripe secret key — MVP workers never touch it
- Used for session lookup on success page (gets customer email)
- Used by webhook for customer email lookup (fallback)
- NOT used for checkout creation (Payment Link handles that)

## Steps

### 0. Check If Payment Proxy Exists FIRST
Before doing anything, check if `payment-proxy` proxy is already deployed. Run:
```bash
curl -s -o /dev/null -w "%{http_code}" https://payment-proxy.{your-account}.workers.dev/
```
If it returns 200, the proxy exists — skip the proxy deploy step entirely. The HMAC secret should already be in `PAYMENT_HMAC_SECRET` env var. Do NOT run `deploy_worker.py proxy` again unless it actually returns 404.

### 1. Data Intake
Sources: `~/.hermes/data/business-ideas.json`, user description, or competitor analysis.
Extract: value prop, pain point, audience, price (one-time), must-have features.

### 2. Build Market-Testing Worker (Generate + Fill Copy)

**Step 2a: Generate the worker JS:**
```bash
python3 ~/.hermes/skills/business/mvp-storefront/scripts/deploy_worker.py build {script-name} "Product Name"
```

This creates `~/.hermes/mvps/{script-name}/src/worker.js` from templates with basic fields filled. It lists the remaining placeholders you need to fill.

**Step 2b: Fill in ALL copy BEFORE deploying:**

Edit `~/.hermes/mvps/{script-name}/src/worker.js` and replace every `{{PLACEHOLDER}}` with real product copy. Typical placeholders:
- `{{PAIN_1}}`, `{{PAIN_2}}`, `{{PAIN_3}}` — specific pain points
- `{{SOLUTION_INTRO}}` — one-sentence solution
- `{{BENEFIT_1_TITLE}}`, `{{BENEFIT_1_DESC}}` etc — outcomes, not features
- `{{STAT_1_VALUE}}`, `{{STAT_1_LABEL}}` — verifiable facts, not fabricated
- `{{FAQ_1_Q}}`, `{{FAQ_1_A}}` — real objections, honest answers
- `{{CTA_HEADLINE}}`, `{{CTA_SUBTEXT}}` — closing argument

**CRITICAL: Never deploy with unfilled placeholders visible to the public.** Fill everything first.

### 3. Inject Analytics Tracker (BEFORE Deploy)

Every MVP storefront MUST include the ALLMIND analytics tracker before deploying. This gives us click, scroll, and section visibility data from day one.

**Script tag to inject** — add before `</body>` in the landing page HTML (inside the worker.js template literal):
```html
<script src="https://yourdomain.com/src/tracker.js" data-source="{script-name}"></script>
```

Replace `{script-name}` with the actual script name (e.g., `squeezed`, `coven-compass`). This tags all events so we can filter analytics by site.

**Where to inject:**
- If worker.js has inline HTML (template literal): find `</body>` inside the template, insert script tag before it
- If using build script with separate landing.html: add to landing.html before `</body>`, build script will include it
- If multiple HTML templates exist (landing, app, success): inject into ALL of them

**CRITICAL: Script must NOT use `DOMContentLoaded` listener.** The tracker script is placed at the END of the `<body>` — by the time it loads, `DOMContentLoaded` has already fired. Any code wrapped in a `DOMContentLoaded` listener will silently never execute. The tracker code must run directly inside the IIFE, not deferred. This was a real bug found Apr 2026 — 27 ad clicks with zero tracked events before the fix.

**What it tracks:** pageviews, clicks (element + coordinates), scroll depth (25/50/75/100%), section visibility, time on page. Zero PII. Events batch to `yourdomain.com/track` every 10s + on page unload.

**Verify after deploy:**
```bash
curl -s -A "Mozilla/5.0" "https://{script-name}.yourdomain.com/" | grep 'tracker.js'
```

### 5a. Graduating a Custom Market-Testing Worker (Manual Conversion)

**WARNING:** `deploy_worker.py mvp` replaces the ENTIRE worker with `stripe-worker.js` template. If your market-testing worker has custom code (teaser calculator, interactive app, unique landing page) that can't be regenerated from a template, the deploy script will DESTROY it.

**When to use manual conversion instead:**
- Worker has a custom teaser calculator (e.g., squeezed's retirement cost estimator)
- Worker has a `/app` route with an interactive tool
- Landing page has custom JS, animations, or data that took effort to build
- The "product" is the app itself (hosted at `/app`), not a downloadable zip

**Manual conversion steps:**

1. **Create Stripe Payment Link** (via API, using existing price ID from `deploy-metadata.json`):
   ```bash
   curl -s -X POST https://api.stripe.com/v1/payment_links \
     -u "$STRIPE_SECRET_KEY:" \
     -d "line_items[0][price]=price_XXX" \
     -d "line_items[0][quantity]=1" \
     -d "after_completion[type]=redirect" \
     -d "after_completion[redirect][url]=https://{name}.yourdomain.com/success?session_id={CHECKOUT_SESSION_ID}" \
     -d "customer_creation=always"
   ```
   Save the returned Payment Link URL.

2. **Add `SUCCESS_PAGE_HTML` constant** to the worker (before the router):
   ```javascript
   const SUCCESS_PAGE_HTML = `<!DOCTYPE html>...`;
   ```
   Use the `assets/success-page.html` template or copy from `stripe-worker.js`. Include Meta pixel with `Purchase` event.

3. **Add `/success` route** to the router:
   ```javascript
   else if (path === '/success') response = htmlResponse(SUCCESS_PAGE_HTML);
   ```

4. **Wire the buy button** to the Payment Link URL:
   - Find the buy button in the landing page HTML (may be `href="#"` or an email-capture form)
   - Replace with `<a href="https://buy.stripe.com/...">` or set via JS: `document.getElementById('buyBtn').href = 'https://buy.stripe.com/...'`
   - Update price text to match actual Stripe price

5. **Add Meta pixel** to landing page (`PageView`) and success page (`Purchase` with value)
   - The deploy script auto-extracts the pixel ID from `fbq('init', '...')` in worker.js and saves it to `deploy-metadata.json` as `meta_pixel_id`

6. **Deploy** with wrangler (NOT the deploy script — you want to preserve custom code):
   ```bash
   cd ~/.hermes/mvps/{name} && npx wrangler deploy
   ```
   Or use `deploy_worker.py mvp` ONLY if you pass the fully-custom worker.js as the source (the script will still create Stripe infra but upload your custom file).

**Verify after deploy:**
- Landing page: `curl -s -A "Mozilla/5.0" https://{name}.yourdomain.com/ | grep -o "buy.stripe.com"`
- Success page: `curl -s -A "Mozilla/5.0" https://{name}.yourdomain.com/success | grep -o "fbq('track', 'Purchase'"`

## Landing Page Pattern: Teaser Calculator → Direct Buy

For calculator/analysis tools, skip the email capture entirely. Show a teaser result that sells the full tool.

**The flow:**
1. Landing page has a simple calculator (3-5 inputs max — just enough to produce a compelling number)
2. User clicks "Calculate" → result shows IMMEDIATELY (no email gate)
3. Below the result: social proof copy that scales with their input ("Users in similar situations save an avg of $X/year")
4. Below that: Stripe Payment Link buy button ("Get Full Analysis — $29")
5. No email collection on the landing page — Stripe captures email at checkout

**Why this works better than email capture:**
- Email gates kill momentum — they've already invested effort, don't want another friction step
- The teaser number creates desire for the full analysis — you don't need to capture them, they WANT to buy
- Stripe collects email at checkout — one collection point, zero landing page friction
- The teaser result IS the ad — people screenshot and share their number

**Social proof copy formula:**
```javascript
var avgSavings = Math.round(total * 0.35); // 35% of their calculated cost
document.getElementById('savingsText').textContent =
  'Users in similar situations find an average of $' + avgSavings.toLocaleString() +
  '/year in savings after taking action.';
```
The number scales with their input, so it always feels relevant.

**Implementation:**
1. Build teaser HTML with inputs + calculate button + hidden result div
2. JS: on click, calculate, show result div, hide input div
3. Result div contains: the number, breakdown, savings copy, `<a>` to Payment Link
4. Payment Link `after_completion` redirect → `/app` (the full tool)
5. No `/signup` route needed — can remove email KV if switching from market-testing

**When to use:** Calculator tools, analysis tools, estimators — anything where a partial result creates desire for the full result.
**When NOT to use:** Physical products, services requiring consultation, products where the teaser IS the product.

## Freemium Tab-Based App Pattern

For hosted web apps with multiple features (e.g., lookup tools, trackers, calculators), make the core feature free and gate premium features behind a per-tab paywall. This lets visitors experience value immediately while creating natural upgrade moments.

**When to use:**
- The product is a multi-feature web app (not a single calculator)
- One feature is a "hook" that demonstrates value in 10 seconds
- Premium features are clearly additive (history, analysis, reverse search)
- You want organic sharing — free users can use the tool and share it

**When NOT to use:**
- The free feature is so complete that nobody needs premium
- Premium features feel like "paying to fix a broken free version"
- The app only has one screen (use teaser calculator instead)

### Architecture

```
Tab 1: Core Feature (FREE) — e.g., "By Intention" lookup
Tab 2: Premium Feature A (PAID) — e.g., "Reverse Lookup"
Tab 3: Premium Feature B (PAID) — e.g., "What Do I Have"
Tab 4: Premium Feature C (PAID) — e.g., "Spell Log"
```

Each paid tab has a CSS paywall overlay inside its panel. The overlay is `position: absolute` within the panel (which is `position: relative`) so it only covers that tab's content, not the whole page.

**3. Unlock check (runs on load + after tab switch):**

### Why This Works

- **Immediate value:** The free tab proves the product works in under 10 seconds
- **Feature-specific upsell:** Each paywall explains exactly what that tab does, not generic "upgrade" copy

- **Two unlock paths:** localStorage for normal use, URL param for email recovery or incognito mode

## Retrofit Persuasion Funnels onto Existing MVPs

When an existing deployed MVP has good ad CTR (4%+) but zero conversions, the product works — the funnel doesn't. Instead of rebuilding, surgically add persuasion layers on top. This is faster than a rebuild and preserves all existing value.

### When to Use
- Autopsy shows "no landing page" or "teaser gives away punchline" failures
- Product has real utility but visitors leave before buying
- You want to fix conversion without touching the core app logic
- Existing worker already serves the product at `/` directly

### Pattern: Landing Page → App Split

**Before:** `GET /` → serves the full app/calculator directly. No persuasion, just tool.
**After:** `GET /` → landing page with teaser + CTA. `GET /app` → existing app (unchanged).

**Steps:**

1. **Read existing worker.js** — map routes, find where product HTML lives (`APP_HTML`, etc.)

2. **Build LANDING_PAGE_HTML constant** — insert BEFORE the existing `APP_HTML` in worker.js:
   - Hero headline naming the pain/outcome (not describing the product)
   - 3 pain cards (daily annoyance, hidden cost, emotional toll)
   - Teaser calculator or demo that shows what they know but blurs what they don't
   - Social proof section with stats + testimonials
   - CTA to existing Stripe Payment Link
   - Email capture form for retargeting (if SIGNUPS KV exists)
   - Meta pixel events: PageView, ViewContent, InitiateCheckout, Lead

3. **Update router** — change `GET /` from `APP_HTML` to `LANDING_PAGE_HTML`:
   ```javascript
   // Before: if (path === '/') response = htmlResponse(APP_HTML);
   // After:  if (path === '/') response = htmlResponse(LANDING_PAGE_HTML);
   ```

4. **Add `/app` route** — if not already present, inject before the 404 fallback:
   ```javascript
   else if (path === '/app') response = htmlResponse(APP_HTML);
   ```

5. **Update success page redirects** — change from `/` to `/app`:
   - `window.location.href = '/'` → `'/app'`
   - Email link `${env.BASE_URL}/` → `${env.BASE_URL}/app`

6. **Add "Try Free" link** on landing CTA section:
   ```html
   <p style="font-size: 12px; color: var(--muted); margin-top: 8px;">
     <a href="/app">Try the free calculator</a>
   </p>
   ```

7. **Add InitiateCheckout tracking** on buy buttons:
   ```html
   <a href="PAYMENT_LINK" onclick="fbq('track', 'InitiateCheckout');">Buy — $12</a>
   ```

8. **Validate + deploy:**
   ```bash
   node -c src/worker.js  # syntax check
   python3 deploy_worker.py mvp {name} src/worker.js "Product" 1200
   curl -s -H "User-Agent: Mozilla/5.0" https://{name}.yourdomain.com/ | grep -o 'tracker.js\|buy.stripe.com'
   ```

Each page in the MVP MUST have a complete pixel block — not just one event. Missing `init` means all tracking silently fails.

**Required per page:**

### Source Code Recovery: You Can't Get It Back from Cloudflare
The Cloudflare API has two endpoints for worker code:
- **Current live worker:** `GET /workers/scripts/{name}` — returns the full multipart source. This works.
- **Version list:** `GET /workers/scripts/{name}/versions` — returns metadata (version IDs, timestamps) but NOT the actual script content. The `/content/v-zip` endpoint returns 405 or JSON errors for version-specific downloads.

**You CANNOT recover a lost deployment from Cloudflare's API.** Historical versions are metadata-only. If you lose your source code, it's gone forever.

**Mandatory recovery protocol:**
1. **Always commit before every deploy.** `git add src/worker.js && git commit -m "..." && python3 deploy-copy.py` — never upload without a commit.
2. **Never `git reset --hard` without knowing exactly which commit you're reverting to.** When told to "revert to this morning's version," verify the commit date with `git log --date=local` BEFORE resetting.
3. If you DO break a deployment and have no git commit, the ONLY recovery path is the `GET /workers/scripts/{name}` multipart extract (current live). Historical versions are not retrievable.

### Common Pitfalls
- **Forgetting to update success page redirect** — After purchase, user lands on `/` (the new landing) instead of `/app`. Always check `window.location.href` in the success page script.
- **Email link points to wrong URL** — Post-purchase email should link to `/app`, not `/`. Update `sendAccessEmail()` function.
- **Duplicate CSS classes across pages** — Landing page and app have separate `<style>` blocks. Classes don't share. If you add a `.cta-btn` class to the landing, it won't exist in the app (and vice versa).
- **Teaser gives away too much value** — The teaser should create desire for more, not satisfy curiosity. Show the input number, blur the insight number.

## Conversion Diagnosis: Good CTR, Zero Purchases

When ads drive clicks (CTR 2%+) but nobody buys, the problem is the landing page, not the ad. Two common failure modes and their fixes, proven Apr 2026 on squeezed and coven-compass:

### Failure Mode 1: Teaser Gives Away the Punchline (Calculator Tools)
**Symptom:** User interacts with calculator, sees a compelling number, leaves. High engagement, zero conversion.
**Root cause:** The teaser result IS the product. After seeing "$40,500/year," the user thinks "OK I know the problem" and doesn't need to pay.
**Fix:** Show the INPUT number (what they already know — their spending) but LOCK the INSIGHT number (the derived value they don't know — retirement cost, runway, etc.). Use CSS `filter: blur(8px)` with a lock overlay. The blurred number creates desire; the lock creates urgency.
**Implementation:**
- Show: "$40,500/year going to dependents" (familiar, not shocking)
- Blur: "$1,017,725" (compounding retirement cost — the money shot)
- CTA: "See Your Retirement Cost — $29"
- Copy: "Every year you wait costs you exponentially more"

### Failure Mode 2.1: Good Ad CTR, Zero Purchases - Missing Interactive Preview (Proven May 2026)
When an ad has good CTR but zero purchases, visitors arrive, read the text, and leave. They can't visualize the product. Two levels of "show don't tell" matter:

**Level 1 - Static mockup (minimum):** A browser-frame mockup showing the tool interface with sample data. This worked initially but conversion still flatlined.

**Level 2 - Interactive preview (proven fix):** Replace the static mockup with a live, functional preview of the product at the top of the page. An actual working dropdown/selector that lets visitors experience 3-4 of the 7 categories instantly, with the remaining categories **blurred behind a frosted-glass overlay with a lock icon and "Unlock" call-to-action**. This creates an immediate "taste + tease" loop:
- Pick an intention → get instant results (first 3 categories clear)
- See there are 4 more but they're blurred and out of reach (filter: blur(3px), opacity: 0.15)
- A lock button overlaid on the blurred section says "Unlock 4 more · $12 once" → links to Stripe

The blur technique is more effective than hiding the categories entirely. Visible-but-unreachable content triggers the Zeigarnik effect (unfinished things nag at you) far more than hidden content (which is easy to scroll past).

**Placement:** Make the interactive preview the very first thing on the page — hero section goes below it. Users should get value immediately without scrolling.

**Copy that converts:** "Pick an intention. See your 3 categories free." + "This is just 3 of 7 categories. Get the full set — and the spell tracker — for $12 once."

### Failure Mode 2: All Tell, No Show (Tool/Demo Products)
**Symptom:** Great ad copy, good CTR, user reads the whole page, leaves. They believe the product exists but can't visualize using it.
**Root cause:** Landing page describes features with words but never SHOWS the tool. No screenshot, no mockup, no demo. The user can't answer "what does this look like when I use it?"
**Fix:** Add a CSS-only mockup of the tool interface between the Solution and Proof sections. Show the tool in action with realistic input/output — e.g., "protection" typed in with all 7 results visible. No interactivity needed; a static mockup that looks like the real thing is enough.
**Implementation:**
- Browser-chrome frame (3 colored dots, app name)
- Search/input bar with realistic query
- Results grid showing the tool's output
- Caption reinforcing the value: "All seven categories, one screen, instant."

### General Conversion Checklist
Before blaming the ad for poor conversion, verify:
- [ ] Buy button is visible ABOVE THE FOLD or immediately after the value moment (not buried)
- [ ] Payment Link actually works (click through to Stripe checkout)
- [ ] Price is justified by the teaser (if teaser gives everything away, price feels like a scam)
- [ ] User can visualize the product (screenshot, mockup, or interactive demo)
- [ ] No unnecessary friction between interest and purchase (email gates, multi-step flows)

## Pitfalls
- **Tracker script must NOT use DOMContentLoaded** — The tracker loads at end-of-body (`</body>`) where `DOMContentLoaded` has already fired. Using `document.addEventListener('DOMContentLoaded', callback)` means the callback never runs and zero events are tracked. Run tracker code directly (no event listener wrapper). Verified broken Apr 2026 — cost a full day of zero tracking data across all MVPs.
- **Analytics tracker must be injected before deploy** — Every MVP storefront needs `<script src="https://yourdomain.com/src/tracker.js" data-source="{script-name}"></script>` before `</body>` in ALL HTML templates. Without this, we get zero behavioral data from ad traffic. Verify with `curl -s -A "Mozilla/5.0" https://{name}.yourdomain.com/ | grep tracker.js` after deploy. The tracker sends events to `yourdomain.com/track` — CORS is locked to *.yourdomain.com domains only.
- **Payment Link `customer_creation` MUST be `'always'`** — default is `'if_required'` which means no customer is created, `session.customer` is null, email lookup fails, success page shows manual entry, webhook can't send email. Cannot be changed after creation — must create new Payment Link.
- `env` only available inside Worker fetch handlers, not at module top-level
- Stripe webhook is auto-registered by the deploy script — no dashboard step needed
- KV namespace must exist before binding to worker
- Workers.dev blocks bare urllib/curl (error 1010) -- use browser User-Agent for testing. ALSO: custom domains (yourdomain.com) return 404 to bare curl even when the route works. Always test with `-H "User-Agent: Mozilla/5.0"`. Example: `curl -s -H "User-Agent: Mozilla/5.0" https://product.yourdomain.com/app`
- Workers.dev subdomain DISABLED by default -- must POST to enable or you get "nothing here"
- Worker upload is multipart/form-data, NOT JSON
- DO NOT use wrangler -- all deploys via Cloudflare REST API
- One-time purchases: no recurring param on the price, checkout mode is "payment" not "subscription"
- ALWAYS update business-ideas.json after deploy -- the pipeline depends on it
- **GET /workers/scripts/{name} returns raw multipart binary, NOT JSON** — cannot parse as JSON. Check HTTP status only (200 = exists, 404 = not found). Do NOT attempt json.loads() on this response.
- **KV namespace creation returns 400 if already exists** — the deploy script must handle this gracefully (skip or use existing ID), not crash. If the script fails on existing KV, do a manual deploy: create Stripe price, read worker JS, multipart upload with existing KV IDs. See `~/.hermes/mvps/{name}/deploy-metadata.json` for existing IDs.
- **Custom domains need 3 token permissions**: Zone → DNS → DNS → Edit (for CNAME), Zone → Workers Routes → Edit (for route), Zone → Zone → Zone → Read (for zone listing). All three required.
- **Worker source download includes multipart boundaries** — if fetching worker source via GET, strip the multipart headers before re-uploading. Extract content between blank line after `name="worker.js"` and end of response.
- **Workers.dev blocks bare curl (error 1010)** — testing deployed workers with bare `curl` returns 404 or 1010 error because Cloudflare blocks requests without a browser User-Agent. Use `curl -H "User-Agent: Mozilla/5.0"` for testing. Browser access works fine.
- **Image embedding in workers**: Base64-encoded JPEGs inline in worker JS. ~140KB JPEG → ~184KB base64 → ~64KB gzipped. One hero image is fine. Multiple images will bloat the worker beyond comfort. For secondary images, use CSS-only visuals (gradients, shapes, emojis). Deploy script has `extra_files` param but it only handles JS modules, not static assets.
- **Duplicate code on re-deploy** — when patching worker source downloaded from CF, watch for duplicate function declarations. CF validates syntax on upload and rejects duplicates.
- **Meta Pixel ID must be real in worker.js, not `{{META_PIXEL_ID}}`** — The deploy script reads your worker.js verbatim and uploads it. It does NOT template-replace `{{META_PIXEL_ID}}` placeholders. If you pass `assets/stripe-worker.js` directly, the deployed page will contain the literal string `{{META_PIXEL_ID}}` and tracking won't fire. Replace with your real pixel ID (e.g., `YOUR_META_PIXEL_ID`) in the worker.js BEFORE deploying. The script then auto-extracts the ID via regex and saves it to `deploy-metadata.json`.
- **Custom workers DON'T get payment routes auto-wired** — The deploy script creates Stripe/KV/webhook infrastructure but uploads whatever worker.js you give it. If you pass a custom worker (not stripe-worker.js template), you MUST manually add: (1) proxy client functions (`signPayload`, `callProxy`), (2) webhook handler (`handleWebhook` for `checkout.session.completed`), (3) email sender (`sendAccessEmail` via Resend), (4) the `/webhook` route in the router. Without these, payments work but no email fires and webhook events are silently dropped. Verify with `grep -n handleWebhook your-worker.js` before deploying full mode on a custom worker.

- **CSS classes in multi-page inline workers must be duplicated per page** — When a worker contains multiple HTML template literals (e.g., `APP_HTML`, `SUCCESS_HTML`, `LANDING_HTML`), each page has its own isolated `<style>` block. CSS rules are NOT shared across pages. If you add a `.paywall-overlay` class to APP_HTML's markup but only define the `.paywall-overlay { ... }` rule in SUCCESS_HTML's `<style>`, the app page renders the element as unstyled (static block instead of fixed overlay). Always verify CSS rules exist in the `<style>` block of EVERY HTML constant that uses those classes.
- **JavaScript escaping in embedded `<script>` tags inside template literals** — `worker.js` contains HTML as a template literal (backticks), and inside that HTML are `<script>` tags with JavaScript code. `node -c` validates the OUTER worker.js file, but the browser parses the INNER script content separately. If you write `'We\'ll'` inside the template literal, Node sees a valid escaped apostrophe, but when served to the browser the backslash may be consumed by the template literal parsing, leaving `We'll` which breaks the single-quoted string. **Fix: Always use double quotes for strings containing apostrophes inside template literals.** Example: `"Saved. We'll send you a summary."` instead of `'Saved. We\'ll send you a summary.'`. After any edit to embedded script content, extract the `<script>` block and validate it separately: `curl -s https://{name}.yourdomain.com/ | grep -oP '(?<=<script>).*?(?=</script>)' > /tmp/script.js && node -c /tmp/script.js`.
- **"Blank page" = JavaScript error in embedded script, not missing HTML** — When a worker app uses CSS to hide non-active screens (`display: none` on `.screen` except `.active`), a JS syntax error in the embedded `<script>` means `init()` never runs, so no screen gets the `active` class, and the page appears completely blank even though all HTML is present in the DOM. **Debugging workflow:** (1) `browser_console` check for JS errors, (2) `document.querySelectorAll('.screen').length` to verify HTML exists, (3) extract `<script>` content via curl and validate with `node -c`, (4) fix escaping issues, (5) redeploy. Do NOT assume the worker failed to deploy — `node -c` on worker.js may pass while the embedded script is broken.
- **Fixed overlays need scroll lock on both `<body>` and `<html>`** — A `position: fixed; inset: 0` overlay covers the viewport but does NOT prevent scrolling. If the page content exceeds viewport height, the user can scroll down and see unblurred content below the overlay. Lock scrolling in JavaScript: `document.body.style.overflow = 'hidden'; document.documentElement.style.overflow = 'hidden';` and restore with `''` when the overlay closes.
- **Route creation 400 errors fail silently** — Fixed in script (Apr 2026). The script now lists existing routes before creating, updates via PUT if route exists, and exits non-zero with full error body on failure. Previously a 400 would print a soft warning and continue, leaving the site unreachable (522) with no indication of why. If you see a 522 after deploy, check that the route exists: `GET /zones/{ZONE_ID}/workers/routes`

## Subdomain Isolation & Security

### Credential Isolation (CRITICAL)
- Each MVP worker gets its OWN scoped secrets -- never share STRIPE_SECRET_KEY across workers
- Create a dedicated Stripe restricted key per MVP: Dashboard > Developers > API Keys > Restricted Keys
  - Scope to only: Charges (write), Checkout Sessions (write), Products (read), Webhooks (write)
  - NEVER use the account-wide secret key for production workers
- CLOUDFLARE_TOKEN should be scoped per-worker if possible (Workers Scripts:Edit on specific scripts only)
- Store per-MVP secrets in `business-ideas.json` under each idea, NOT in shared .env

### Cookie Scoping (CRITICAL)
- ALL cookies MUST be set with `Domain={specific-subdomain}.yourdomain.com` -- NEVER `.yourdomain.com`
- All cookies must include `Secure; HttpOnly; SameSite=Strict` flags
- Session/auth cookies must be scoped to the exact subdomain, not the apex domain
- Example correct Set-Cookie header:
  ```
  Set-Cookie: session=abc123; Domain={script-name}.yourdomain.com; Secure; HttpOnly; SameSite=Strict; Path=/
  ```

### Worker Isolation Rules
- Each MVP = one dedicated Worker (no shared workers across products)
- Each MVP = its own KV namespaces (customers + sales tracking) -- never share KV across MVPs
- Each MVP = its own Stripe restricted key -- never share Stripe keys across MVPs
- If one worker is compromised, blast radius = that one product only

### Security Checklist (Post-Deploy)
- [ ] Stripe key is restricted (not account-wide sk_live_)

## Landing Page Pattern: Teaser Calculator → Direct Buy

For calculator/analysis tools, skip the email capture entirely. Show a teaser result that sells the full tool.

**The flow:**
1. Landing page has a simple calculator (3-5 inputs max — just enough to produce a compelling number)
2. User clicks "Calculate" → result shows IMMEDIATELY (no email gate)
3. Below the result: social proof copy that scales with their input ("Users in similar situations save an avg of $X/year")
4. Below that: Stripe Payment Link buy button ("Get Full Analysis — $29")
5. No email collection on the landing page — Stripe captures email at checkout

**Why this works better than email capture:**
- Email gates kill momentum — they've already invested effort, don't want another friction step
- The teaser number creates desire for the full analysis — you don't need to capture them, they WANT to buy
- Stripe collects email at checkout — one collection point, zero landing page friction
- The teaser result IS the ad — people screenshot and share their number

**Social proof copy formula:**
```javascript
var avgSavings = Math.round(total * 0.35); // 35% of their calculated cost
document.getElementById('savingsText').textContent =
  'Users in similar situations find an average of $' + avgSavings.toLocaleString() +
  '/year in savings after taking action.';
```
The number scales with their input, so it always feels relevant.

**Implementation:**
1. Build teaser HTML with inputs + calculate button + hidden result div
2. JS: on click, calculate, show result div, hide input div
3. Result div contains: the number, breakdown, savings copy, `<a>` to Payment Link
4. Payment Link `after_completion` redirect → `/app` (the full tool)
5. No `/signup` route needed — can remove email KV if switching from market-testing

**When to use:** Calculator tools, analysis tools, estimators — anything where a partial result creates desire for the full result.
**When NOT to use:** Physical products, services requiring consultation, products where the teaser IS the product.

## Enriching Paid Content with Interactive Tools

When an MVP storefront has a paywalled app (not a downloadable zip), the $29 unlock needs to feel like a toolkit, not a PDF. Adding interactive screens — sliders, toggle cards, live dashboards — significantly increases perceived value and conversion.

### When to Add Interactive Tools
- Teaser calculator is live but paywall conversion is low
- User feedback says "I want to play with the numbers"
- The paid product feels like "just text" (scripts + bar charts)
- You want differentiation from free blog posts on the same topic

### Screen Architecture Pattern
For a paywalled app with multiple screens, use this structure:
```

Screen 2: Input form (free)
Screen 3: Core insight with paywall overlay (free preview, locked detail)
Screen 4: Boundary plan / scripts (paid)
Screen 5: Interactive Tool 1 — Transition Lab (paid)
Screen 6: Interactive Tool 2 — Support Mix (paid)
Screen 7: Save & Share / Summary (paid)
```

### Adding New Screens to an Existing App

**1. Renumber carefully.** If inserting screens before the final screen, rename the old final screen to avoid ID conflicts:
```javascript
// Old: screen4 → plan, screen5 → save
// New: screen4 → plan, screen5 → tool1, screen6 → tool2, screen7 → save
```
Update ALL references: `id="screenN"`, `goTo(N)`, progress dots `data-step="N"`.

**2. Update progress dots.** Add new `<div class="progress-dot" data-step="N">` elements for each new screen.

**3. Expand paywall scope.** The `goTo` function must block ALL new paid screens:
```javascript
var origGoTo = goTo;
goTo = function(screen) {
  if ((screen === 4 || screen === 5 || screen === 6 || screen === 7) && !checkUnlock()) {
    goTo(3);
    applyPaywallState();
    return;

  origGoTo(screen);
  if (screen === 5) setTimeout(updateTransition, 100);
  if (screen === 6) setTimeout(updateMix, 100);
};
```
**Critical:** Capture `origGoTo` BEFORE overriding. If you call `goTo(3)` inside the override without the original reference, you get infinite recursion.

**4. Update paywall feature list.** The overlay's bullet list should call out the new tools:

## Print Summary for Multi-Screen Apps

When a paywalled app has multiple screens (`display: none` / `display: block` toggle), adding a "Print" button seems simple: `@media print { .screen { display: block !important } }`. **This fails.** The printed page is blank or broken because:
- Screens 5-N only populate data when visited; unvisited screens print empty
- CSS `!important` overrides fight with the app's active-screen logic
- Layouts designed for viewport stacking don't paginate

### Working Pattern: Dedicated Print Container

Instead of forcing visible screens, create a hidden single-page summary that JavaScript populates:

**1. Add a hidden print container in the HTML:**
```html
<div id="printSummary" class="print-summary" style="display: none;">
  <h1>Your Analysis Summary</h1>
  <p class="print-date"></p>
  <div class="print-section">
    <h2>Your Situation</h2>
    <p>Age: <span class="print-age"></span></p>
    <p>Annual support: <span class="print-support"></span></p>
  </div>
  <!-- ... more sections ... -->
</div>
```

**2. Add print CSS that hides everything except the summary:**
```css
@media print {
  .no-print { display: none !important; }
  body { background: white; color: black; }
  #printSummary {
    display: block !important;
    position: static;
    width: 100%;
    padding: 20px;
  }
  .screen, .progress-bar, .paywall-overlay, .back-link {
    display: none !important;
  }
}
```

**3. Add a `printSummary()` function:**
```javascript
function printSummary() {
  var r = results; // global results object from calculator
  if (!r || !r.fvDependentMoney) {
    alert('Please complete the analysis first.');
    return;
  }
  var d = loadData(); // form inputs
  document.querySelector('.print-date').textContent = new Date().toLocaleDateString();
  document.querySelector('.print-age').textContent = d.age;
  document.querySelector('.print-support').textContent = '$' + d.annualSupport.toLocaleString();
  // ... populate all .print-* elements ...
  window.print();
}
```

**4. Wire the print button:**
```html
<button onclick="printSummary()">Print This Summary</button>
```

**Why this works:**
- The summary is always in the DOM, so `@media print` doesn't fight JS state
- JavaScript guarantees data exists before printing (no blank sections)
- Single clean layout designed for paper, not screen stacking
- No `!important` wars with `.active` screen classes

### Common Print Pitfalls
- **Never use `window.print()` directly** on a multi-screen app without populating a print container first — you'll print whatever screen happens to be visible, which may be incomplete or unreadable
- **Always validate `results` before printing** — if the user hasn't run the calculator, show an alert instead of a blank page
- **Test with real browser print preview** — `curl` can verify elements exist but can't render print CSS; open the page in a real browser and use Ctrl+P

## Back Navigation for Multi-Screen Apps

When a paywalled app has 4+ screens, users expect to move backwards. Without back navigation, the app feels like a linear funnel and users get trapped — especially on mobile where the browser back button reloads the page and resets state.

### Pattern: Clickable Progress Dots + Back Links

**State tracking:**
```javascript
var maxVisitedScreen = 1; // tracks how far user has gone

function goTo(screen) {
  // paywall check...
  document.querySelectorAll('.screen').forEach(function(s) { s.classList.remove('active'); });
  document.getElementById('screen' + screen).classList.add('active');
  document.querySelectorAll('.progress-dot').forEach(function(d) {
    var step = parseInt(d.dataset.step);
    d.classList.remove('active', 'done', 'clickable');
    if (step === screen) d.classList.add('active');
    else if (step < screen) d.classList.add('done');
    if (step <= maxVisitedScreen && step !== screen) d.classList.add('clickable');
  });
  if (screen > maxVisitedScreen) maxVisitedScreen = screen;
  currentScreen = screen;
  window.scrollTo(0, 0);
}
```

**CSS for clickable dots:**
```css
.progress-dot {
  width: 8px; height: 8px; border-radius: 50%;
  background: var(--cream-dark); transition: background 0.3s;
  cursor: default;
}
.progress-dot.clickable {
  cursor: pointer;
}
.progress-dot.clickable:hover {
  background: var(--gold);
  transform: scale(1.3);
}
.progress-dot.active { background: var(--gold); }
.progress-dot.done { background: var(--stone-light); }
```

**Wiring clicks in init():**
```javascript
document.querySelectorAll('.progress-dot').forEach(function(d) {
  d.addEventListener('click', function() {
    var step = parseInt(d.dataset.step);
    if (step <= maxVisitedScreen) goTo(step);
  });
});
```

**Back links on each screen:**
```html
<!-- Screen 2 -->
<button class="back-link" onclick="goTo(1)">← Back</button>

<!-- Screen 3 -->
<button class="back-link" onclick="goTo(2)">← Back</button>
```

```css
.back-link {
  display: inline-flex; align-items: center; gap: 6px;
  font-size: 13px; color: var(--stone);
  cursor: pointer; transition: color 0.2s;
  background: none; border: none; padding: 0;
  font-family: inherit; margin-bottom: 16px;
}
.back-link:hover { color: var(--gold); }
```

**Behavior:**
- Dots for unvisited screens: non-clickable (default cursor)
- Dots for visited screens: clickable with hover effect
- Back buttons on Screens 2-N always work (they go to the previous screen)
- Forward movement past paywall still requires unlock
- `maxVisitedScreen` is session-only (not persisted to localStorage) — resets on page reload, which is fine for navigation state

### Why This Matters
- **Mobile users:** Browser back button reloads the page and loses calculator inputs. In-app back navigation preserves state.
- **Comparison shoppers:** Users want to re-read Screen 3 results after seeing Screen 4's plan. Trapping them forward kills conversion.
- **Trust signal:** Back navigation makes the app feel like a real tool, not a trap.

## Post-Launch Price Changes

Stripe prices are **immutable**. You cannot update an existing price's amount. When lowering (or raising) the price on a live MVP, follow this sequence exactly:

### 1. Create a New Stripe Price
```bash
curl -s -X POST -u "$STRIPE_SECRET_KEY:" https://api.stripe.com/v1/prices \
  -d "product=prod_XXX" \
  -d "unit_amount=1200" \
  -d "currency=usd"
```
Returns: `price_NEWID` at $12.00.

### 2. Create a New Payment Link
```bash
curl -s -X POST -u "$STRIPE_SECRET_KEY:" https://api.stripe.com/v1/payment_links \
  -d "line_items[0][price]=price_NEWID" \
  -d "line_items[0][quantity]=1" \
  -d "after_completion[type]=redirect" \
  -d "after_completion[redirect][url]=https://{name}.yourdomain.com/success?session_id={CHECKOUT_SESSION_ID}" \
  -d "customer_creation=always"
```
Returns: new Payment Link URL (e.g., `https://buy.stripe.com/cNi28r...`).

### 3. Update ALL Price References in worker.js
Search for every occurrence of the old price and replace:
- Paywall display price: `<div class="price">$29</div>` → `$12`
- CTA button text: `'Unlock Full Analysis — $29'` → `'Unlock Full Analysis — $12'`
- Meta Pixel value: `{value: 29.00, currency: 'USD'}` → `{value: 12.00, currency: 'USD'}`
- Payment Link URL: `window.open('https://buy.stripe.com/OLD...')` → `window.open('https://buy.stripe.com/NEW...')`
- Any other copy mentioning the price (headlines, subtext, etc.)

Use `grep -n '\$29\|29\.00\|buy\.stripe\.com/OLD'` to find all references.

### 4. Deploy
Run the standard deploy. The script reuses the existing Stripe product but does NOT auto-update the price — you've already created the new price manually.

### 5. Verify
- Paywall shows the new price
- Payment Link charges the new amount (check via Stripe Dashboard or API)
- Meta Pixel Purchase event fires with the new value
- Old Payment Link still works but is no longer referenced (customers with old links can still buy at old price — archive it if you want to prevent this)

**Pitfall:** The deploy script's output may still show the OLD `stripe_price_id` from `deploy-metadata.json`. This is cosmetic — the worker.js you uploaded contains the new Payment Link URL. The price ID in metadata is only used if the script creates a new price (which it skips on redeploy). To keep metadata accurate, manually update `deploy-metadata.json` with the new price and payment link IDs.

## Lessons Learned (Coven Compass Launch)

### Orphan Stripe Products — FIXED
The deploy script previously created a new Stripe product+price on every run. If you deployed twice, you got two orphan products.
**Fix:** The `deploy_mvp()` function now checks `deploy-metadata.json` for existing `stripe_product_id` and `stripe_price_id`. If found, it reuses them instead of creating new ones. On first deploy, it creates + saves the IDs. On redeploys, it skips creation.
**If you hit orphan products:** Check Stripe dashboard, delete extras. The Payment Link is tied to the original price ID — as long as you didn't delete that, it still works.

### Two-Step Deploy for Payment Link
The Payment Link URL must be in the landing page HTML BEFORE deploy, but you can't create the Payment Link until you have a price ID from deploy. The workaround:

```bash
# Step 1: Deploy first (creates product+price, leaves {{PAYMENT_LINK_URL}} placeholder)
python3 deploy_worker.py mvp ... 500

# Step 2: Create Payment Link via API using the price ID from step 1
curl https://api.stripe.com/v1/payment_links \
  -u sk_...: \
  -d "line_items[0][price]=price_..." \
  -d "after_completion[type]=redirect" \
  -d "after_completion[redirect][url]=https://{name}.yourdomain.com/success?session_id={CHECKOUT_SESSION_ID}" \
  -d "customer_creation=always"

# Step 3: Fill the URL into landing.html, rebuild, redeploy
# (redeploy reuses the product+price from step 1)
```

### Product Packaging Pattern
For downloadable HTML-based products, package as zip and embed in worker:
```bash
# 1. Create zip
zip -j product.zip path/to/product.html

# 2. Convert to base64
base64 -w0 product.zip > product.b64

# 3. Embed in worker.js as:
const ZIP_DATA = "base64string..."
```
Keep the zip under 25MB (Cloudflare Worker size limit). For simple HTML tools, this is trivially small.

### Build Script Pattern
Separate source files are easier to edit than inline HTML in a giant worker.js:
```
src/
├── landing.html      # Edit this for copy changes
├── privacy.html      # Edit this for legal updates
├── terms.html        # Edit this for legal updates
├── build_worker.py   # Assembles worker.js from source + template
└── worker.js         # Generated output (don't edit directly)
```
The build script reads the stripe-worker.js template, injects the HTML files as JS constants, adds ZIP_DATA, and writes the final worker.js. Run `python3 src/build_worker.py` before every deploy.

### Hosted Web App > Downloadable Zip
Mobile browsers block JavaScript execution from locally opened HTML files. Android uses a `content://` URI scheme that doesn't support relative file paths or JS execution. iOS similarly sandboxes local files.

**For consumer products, serve the app directly at a URL (e.g., `/app`) instead of packaging as a zip.** The product HTML is served inline via a Cloudflare Worker constant. Works on all devices, no download required. Cost: $0 on Cloudflare's free tier.

If you must deliver a downloadable, make it a **single completely self-contained HTML file** with zero external dependencies. But even this may fail on mobile.

### Support Email
Use `support@yourdomain.com` everywhere. This email should be configured via Resend for your domain. Never check this inbox programmatically (injection risk).

### Downloadable HTML Products Fail on Mobile — Use Hosted App Instead
Mobile browsers (Android Chrome, iOS Safari) block JavaScript execution from locally opened HTML files. Android uses a `content://` URI scheme that doesn't support relative paths or JS. iOS sandboxes local files similarly. **A single-file HTML product fundamentally won't work on mobile when downloaded.**

**Solution: Host it.** Serve the product directly at a URL (e.g., `/app`) on the same worker. After Stripe checkout, redirect to the app. No download, no zip, no email with download link — just open the tool in the browser.

- Same Cloudflare Worker, $0 additional cost
- Works on all devices (phone, tablet, desktop)
- Payment Link `after_completion` redirect → `{BASE_URL}/app`
- Landing page: link buttons to Payment Link, no download language

**Implementation:**
1. Read product HTML and embed as `APP_HTML` constant in worker.js (via build script)
2. Add route: `else if (path === '/app') response = htmlResponse(APP_HTML);`
3. Update landing page copy: "Works Everywhere" instead of "Own It Forever"
4. Create Payment Link with redirect to `/app`
5. Keep the download/zip/email flow only if the product is a non-HTML package (zip of assets, etc.)

For the build script, inject the `/app` route via post-processing since the shared stripe-worker template doesn't include product-specific routes:
```python
# After writing worker.js from template, inject route:
app_route = "      else if (path === '/app') response = htmlResponse(APP_HTML);\n"
final = final.replace(
    "      else response = jsonResponse({ error: 'Not found' }, 404);",
    app_route + "      else response = jsonResponse({ error: 'Not found' }, 404);"
)
```

### Mobile Browsers + Large Inline JS
NOTE: Both inline AND external JS files fail on mobile when opened locally. The `content://` protocol blocks all JS execution. Don't try to fix this with file structure changes — just host the product instead (see above).

For downloadable HTML products:
- Keep data in a separate `data.js` file (e.g. `correspondences.js` that sets `var DB = {...};`)
- HTML loads it via `<script src="correspondences.js"></script>` in the `<head>`
- Zip includes both files together — user unzips and opens the HTML, script loads from same folder
- **Always test on a real mobile device**, not just desktop

### Product Zip Structure

The Resend email `subject` should include the product name so customers recognize it in their inbox. NOT just "Your download is ready" — use "Your {Product Name} download is ready". Also set `from` as `"{Product Name} <noreply@yourdomain.com>"` so the sender is recognizable.

### Orphaned Code Between Template Literal Boundaries in Workers
When a worker file contains multiple HTML templates as string literals (e.g., `LANDING_PAGE_HTML`, `APP_HTML`, `SUCCESS_PAGE_HTML`), code can become stranded between the closing backtick of one template and the `const` declaration of the next. This orphaned code is never executed but creates silent duplicates that confuse maintenance.

**Real bug (Apr 2026, Coven Compass):** During an APP_HTML rewrite, a Meta Pixel block was left orphaned between `APP_HTML`'s closing backtick and `SUCCESS_PAGE_HTML`. Later, when cleaning up "duplicates," the pixel was accidentally removed from the live `/app` route entirely because the orphaned block was the only one inside APP_HTML — the actual pixel had been moved earlier.

**Prevention:** After any major rewrite of a multi-template worker, grep for tracking code and verify count per route:
```bash
# In source: expect 1 pixel per HTML template that needs it
grep -c "YOUR_META_PIXEL_ID" src/worker.js

# On each live route: expect 2 occurrences (script + noscript)
curl -s https://{name}.yourdomain.com/ | grep -c "YOUR_META_PIXEL_ID"
curl -s https://{name}.yourdomain.com/app | grep -c "YOUR_META_PIXEL_ID"
curl -s "https://{name}.yourdomain.com/success?session_id=test" | grep -c "YOUR_META_PIXEL_ID"
```
If counts don't match expectations, a template is missing its pixel or has an orphan.

### Embedding Images in Worker (No Static Upload)
CF Workers multipart upload's `extra_files` only handles JS modules, not images. For hero/feature images:
1. Compress PNGs to JPEG: `img.resize((max_w, int(height * ratio)), LANCZOS)` then `img.save(dst, "JPEG", quality=80, optimize=True)`
2. Target ~1400px wide max, quality 80 → ~100-150KB per image
3. Base64 encode and embed as `data:image/jpeg;base64,...` in the HTML `src` attribute
4. ~138KB JPEG → ~184KB base64 → ~64KB gzipped — well within CF Worker limits
5. Only embed 1-2 key images (hero, etc.). Worker limit is 1MB total (compressed).
6. `escape_js_template(s)` must handle `\\`, `` \` ``, and `${` to prevent JS syntax errors in template literals.

**When to use:** Hero banners, above-the-fold images where you need them to load instantly with the page. CSS-only sections for the rest (gradients, emoji icons, typography-driven design).

**When NOT to use:** Multiple large images, photo galleries, product screenshots. Use external CDN or Cloudflare Images instead.

- **Hero image placeholder never filled**: If `landing.html` uses `HERO_B64_PLACEHOLDER` as the image src, the build script does NOT auto-fill it. You must run: `base64 -w0 assets/images/hero.jpg` and replace the placeholder in `landing.html` before building. Symptom: `<img>` tag renders as broken image on live page. Check with `browser_console`: `document.querySelectorAll('img').length` — if 0, image isn't in DOM. Verify after every landing.html edit that the base64 data URL is still present.

### Landing Page Conversion Hierarchy (Proven May 2026)

When designing a MVP storefront landing page for paid traffic, follow this order of effectiveness (bottom to top):

**1. Light background, dark text > Dark theme for sales pages.** Dark/void themes look cool but convert poorly for cold traffic. Use light parchment (#FAF8F3) with dark text (#1A1A2E) and keep purple/gold accents. Witchy audience wants the aesthetic but needs readability first.

**2. Interactive demo FIRST, hero second.** Place the working product demo at the very top of the page — before the headline, before the hero, before everything. Users should get value within 2 seconds of landing. The hero section goes below it.

**3. "Pick an intention → see 3 categories" pattern.** For data-lookup products, give visitors a dropdown that shows real results. Populate all 7 categories internally but blur the last 4 with `filter: blur(3px)` + `opacity: 0.15` + a frosted overlay. The blurred-but-visible technique creates much stronger urge-to-complete than hiding the categories entirely.

**4. Lock button on the blur panel.** A purple gradient button overlaid on the blurred section with: lock icon, "Unlock X more", "$12 once". Click goes directly to Stripe. Makes the value gap immediately actionable.

**5. Hammer the subscription vs one-time comparison.** Use a side-by-side pricing showdown: "$29/month ($348/year)" vs "$12 once" with a "Save $336+/year" badge. Visitors to witchy/spiritual products are often bleeding money to $30/month Moonly or $29/month Spells8 — this is the single most effective objection handler for conversion.

**6. Direct FAQ answers the subscription fear.** Include an FAQ like "Will there be a subscription later?" with a blunt "No. We build tools, not rent-seekers."

**7. Commit every deploy.** Before running `python3 deploy-copy.py`, always `git add src/worker.js && git commit -m "..."`. Cloudflare's API does NOT store historical versions — if you lose your source and need to revert, only the current live deploy is recoverable via API. The `GET /workers/scripts/{name}/versions/{id}` endpoint returns metadata only, not script content. This caused total source loss on May 5, 2026.

### Inline Data Corruption When Restoring JS
When removing inline data from a `<script>` block and later restoring it, be very careful. The removal can leave orphaned code lines that reference the removed variable. **Always verify the restored code compiles before deploying.** Example of what went wrong:

```javascript
// BEFORE removal (correct):
var DB = {protection: {...}, love: {...}, ...30 intentions...};
// Build reverse index
var revIdx = {};
Object.keys(DB).forEach(function(int) { ... });

// AFTER bad removal (broken - only 1 intention, orphaned code):
var DB = {protection: {...}}    // only 1 of 30!
  var d = DB[int];              // orphaned — 'int' is undefined
  Object.keys(d).forEach(...);  // orphaned — causes SyntaxError
```

The orphaned `var d = DB[int]` block caused `SyntaxError: Unexpected token` at the line where `int` was referenced. The error appeared at a specific line number in the browser but the ROOT CAUSE was the incomplete DB object on the previous line.

**Prevention:** After any removal/restoration of inline data blocks, verify:
1. The data object has all expected entries (check count)
2. No orphaned code between the data declaration and the next valid function
3. Run a syntax check: `node -c product.html` won't work, but evaluating the script content in browser console catches errors
